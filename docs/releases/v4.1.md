# Release Notes: Coach Engine v4.1
**Release Date:** May 22, 2026  
**Environment:** Production & Sandbox/Demo (`coach-engine-demo`)  
**Status:** Stable, Hardened & Deployed  

---

## 🎯 Executive Summary
Version 4.1 introduces a comprehensive architectural hardening of the application's authentication lifecycle, an entirely new color-coded, context-aware security log (`access_logs`), and environment-aware "Circuit Breakers" for external APIs. These changes significantly lower external API overhead and eliminate system fragility caused by network latencies or third-party components during local development.

---

## 🛠️ New Features & Architectural Enhancements

### 1. Enterprise Audit Trail & Security Panel (`access_logs`)
The legacy, flat logging system has been refactored into a relational database-driven, color-coded event log.
* **Relational Join Ingestion:** Data queries have been centralized within `src/database.py` utilizing an optimized PostgREST explicit join (`audit_log_type!al_action_type_id`). This maps raw database transactions to structured application codes (e.g., `USER_LOGIN`, `AUTH_FAILURE`, `SYSTEM_EVENT`).
* **Security Indicators (🚥 Traffic Light System):** The administrative dashboard now visually categorizes event severity instantly:
  * 🔴 **Red (ERROR):** Security anomalies, failed credential attempts, and brute-force indicators.
  * 🟢 **Green (SUCCESS):** Verified, tamper-proof manual logins and seamless automated cookie sessions.
  * 🟠/🔵/🟡 **Amber/Blue/Yellow:** System anomalies, schema CRUD mutations, and administrative overrides (e.g., forced password resets).

### 2. Just-In-Time (JIT) Network Metadata Enrichment
The logging engine (`logger.py`) has been upgraded to capture and inject client-side network context directly into PostgreSQL JSONB payloads synchronously, without degrading rendering performance.
* **Client Identification:** The system now dynamically extracts the client's actual public IP address (via proxy forwarding) and the full User-Agent string.
* **Hybrid Traceability Lifecycle:** For application-level events, metadata is fetched via Streamlit's context layer. For native database mutations (e.g., PostgreSQL Row-Level triggers like `PARTICIPANT_UPDATED`), user-session traceability is strictly maintained in the DB tier even when HTTP network headers are naturally absent.

### 3. Sandbox-Isolation of the Authentication Loop
To prevent asynchronous race conditions between Streamlit's top-to-bottom execution threads and the browser's JavaScript engine, `views/login.py` has been split into a decoupled, two-phase architecture:
* **Phase 1 (Auth Verification):** A strict, isolated execution block against the Supabase Auth API.
* **Phase 2 (State & Cookie Assignment):** An embedded sandbox that silently intercepts transient third-party cookie latency (`streamlit_cookies_controller`). It logs unexpected connection states as a mild `SYSTEM_EVENT` rather than crashing the UI with a fatal execution stack trace for the end-user.

### 4. Environment-Aware API Circuit Breaker (Weather Engine)
A central environment flag (`IS_DEMO`) has been integrated into `src/config.py` and implemented as a strict "Circuit Breaker" inside `src/utils.py`.
* **API Ingestion Protection:** When the app detects a non-production environment (Local or Demo), all external HTTP requests to the Open-Meteo API are completely bypassed. The engine instantly yields a predictable, hardcoded mock layout (Clear skies, 18°C, 2 m/s), completely eliminating the risk of striking rate limits (`429 Too Many Requests`) during aggressive local development.

## 🐛 Bug Fixes & Stability Improvements
* **Resolved "Two-Click Logout" Phenomenon:** Cleaned up overlapping rendering threads in the `app.py` logout listener. By immediately issuing a strict `st.rerun()` upon purging session hashes, the browser-side cache drops instantly, throwing the client back to the login terminal on the first click.
* **Streamlit API Modernization:** Phased out the deprecated internal function `_get_websocket_headers()` in favor of the official, native `st.context.headers` interface, eliminating framework warnings in the server console.
* **`NoneType` Robustness in Core Auth:** Omslutted core cookie evaluations within `app.py` inside a defensive `try-except TypeError` loop. This elegantly manages the microscopic delay while the browser's storage container is establishing itself during cold reloads.
